[FullStack] Assignment - Calculator

Motivation

Scenius is always looking for motivated and skilled software engineers who are not afraid to tackle every aspect of the software stack. For this purpose, we have set up a code challenge that allows a potential colleague to demonstrate these skills.

Assignment Description

Develop a simple online calculator Software as a Service where a mathematical expression can be read via a Single Page Application. This expression should then be sent via a REST request to a self-developed API called Ingest. This API should then publish this calculation request on a Message Bus. A self-developed console application Consumer should be able to read this queue and solve the expression. Then, the Consumer application should save the result in a database. When the result becomes available, the Ingest application should be informed via the message bus. The result should then be sent back to the front-end application via an HTTP GET or websocket.

When the front-end application is opened, all previous calculations (i.e., those stored in the database) should be displayed.

Our Stack

Scenius adheres to the following stack. Although we are not afraid of unfamiliar code, we would appreciate it if the following stack is largely adhered to:

Role Stack

  • Backend: Latest ASP.NET

  • Frontend: Latest Angular

  • Database: SQL Server, PostgreSQL

  • ORM: Entity Framework Core (Mandatory)

  • Message Bus: RabbitMQ or another AMQP server

As a starting point for the assignment, use of our Starter Pack, which includes a scaffolded Angular & .NET project, is optional but not mandatory. It can be found here: Scenius Starter Pack. It is might be outdated using it as inspiration for the structure is recommended.

Requirements

To guide the assignment and to rank the aspects of the assignment that are important for Scenius, we use the MOSCOW method. Focus on delivering the requirements (Musts), and try to deliver optional requirements where possible (Shoulds, Coulds).

  • Angular clients can send new calculation requests/POST to the API.

    • MUST: The SPA has an input field for calculations, and a table with already entered calculations and results from the database.

    • SHOULD: The application has a good and professional look & feel. Existing UI frameworks may be used.

    • COULD: The frontend is tested using an End-to-End testing framework, such as TestCafe or Protractor.

  • The frontend client needs to set up an HTTP connection to get results back.

    • MUST: New calculation results must be retrievable via an HTTP call

    • SHOULD: New calculation results should be automatically fetched and “live” (periodically) refreshed.

    • COULD: New calculation results are live-streamed via an asynchronous backend to frontend channel as soon as they become available via a WebSocket connection.

  • Upon opening the frontend application, all existing calculations from the database should be shown first.

    • MUST: All previously executed results are displayed

    • SHOULD: Executed results can be paginated so the entire database does not need to be queried at once

  • The API should queue this request in an AMQP message bus with which you are most familiar.

    • MUST: New calculations are sent to the message bus

  • The consumer application needs to connect to the message bus and read the calculation requests, compute them, and store the results in the database.

    • MUST: Storing in the database must be done using an ORM. Use the database you are most familiar with, and use Entity Framework Core for storing and retrieving data in the database.

    • COULD: When a calculation is done, the consumer should send the result back via the message bus.

  • All applications need to be scalable. That is, it must be possible to start more than one instance of both the Consumer and the API when one instance can no longer handle the number of incoming messages.

    • MUST: It must be possible to run more than one instance of the consumer application

    • SHOULD: Both the Consumer and API should be able to run with more than one instance at the same time

  • The calculation service only needs to solve simple expressions using:

    • Symbols: * for multiplication, / for division, + for addition, - for subtraction

    • Note: All operators (-+/*) are binary operators, meaning you do not need to account for cases like -5 - 2 (negative numbers do not need to be supported). Parentheses () do not need to be supported.

    • Examples: 1 + 2, 7 * 3 / 2, 0 / 0, 1 / 0, 15 * 2 – 4 / 2

    • MUST: The calculation service returns a random result

    • SHOULD: The calculation service computes the expression according to the above rules, only supporting binary operators and without the use of parentheses.

    • COULD: The calculation service supports the use of parentheses and unary operators.

  • Testing software before it goes into production is of great importance. Adding Unit Tests for certain aspects of the applications is a good addition.

    • COULD: Write one or more unit tests for the backend projects. Use xUnit, or a testing framework of your choice.

    • COULD: Write one or more unit or component tests for the frontend project. Use Karma for this purpose.

  • At Scenius, we use Git to develop software together. Good use of Git would be a nice addition.

    • SHOULD: Submit the assignment with a Git repository.

    • COULD: Use branches and smaller commits so that changes to the codebase are visible over time.

  • For packaging the software, it is now customary to use containers. This can prevent the well-known Works on my machine problems.

    • COULD: The application is buildable as a container. Create a Dockerfile, and use either a docker-compose file or one or more Kubernetes YML definitions to run the app. Scaling the consumer should also be possible via the containerization tool.
